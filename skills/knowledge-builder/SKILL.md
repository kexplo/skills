# Knowledge Builder

저장소의 구현을 분석하여 코딩 에이전트가 효율적으로 참조할 수 있는 지식 베이스를 구축한다.

## 핵심 원칙

- **토큰 효율성**: 모든 문서에 목차(TOC)를 포함하여 에이전트가 필요한 섹션만 선택적으로 읽을 수 있게 한다
- **점진적 구축**: 처음부터 모든 문서를 만들지 않는다. 저장소의 규모와 복잡도에 따라 필요한 문서만 생성한다
- **초기 생성 vs 갱신**: 기존 지식 베이스가 있으면 보충/수정하고, 없으면 새로 구축한다

## 실행 흐름

### 1단계: 현황 파악

대상 디렉토리(기본값: 현재 작업 디렉토리)에서 기존 지식 베이스 존재 여부를 확인한다.

```
확인 항목:
- ARCHITECTURE.md 존재 여부
- docs/ 디렉토리 존재 여부
- 기존 문서의 최신성 (git log와 비교)
```

**기존 지식 베이스가 있는 경우** → 3단계(갱신 모드)로 이동
**없는 경우** → 2단계(초기 구축)로 진행

### 2단계: 저장소 분석

분석은 넓은 곳에서 좁은 곳으로 진행한다. 각 단계에서 발견한 단서를 다음 단계의 탐색 방향에 반영한다.

#### 2-1. 프로젝트 뼈대 파악

**목표**: 이 프로젝트가 무엇이고, 어떤 도구로 만들어졌는지 빠르게 파악한다.

1. 최상위 디렉토리 레이아웃 매핑 (`ls`, 트리 구조 확인)
2. **빌드/패키지 매니저 식별** — 다음 파일들을 탐색하여 사용 중인 도구 체인을 확정한다:
   - Node.js: `package.json`, `pnpm-workspace.yaml`, `yarn.lock`, `bun.lockb`, `.npmrc`
   - Python: `pyproject.toml`, `setup.py`, `requirements.txt`, `Pipfile`, `uv.lock`, `poetry.lock`
   - Go: `go.mod`, `go.sum`
   - Rust: `Cargo.toml`, `Cargo.lock`
   - Java/Kotlin: `build.gradle`, `pom.xml`, `settings.gradle`
   - Ruby: `Gemfile`, `Gemfile.lock`
   - 기타: `Makefile`, `Dockerfile`, `docker-compose.yml`, `Taskfile.yml`, `justfile`
3. **코드 품질 도구 식별** — 린터, 포매터, 타입 체커 설정 파일을 찾는다:
   - 린터: `.eslintrc*`, `ruff.toml`, `.flake8`, `.golangci.yml`, `.rubocop.yml`, `clippy.toml`
   - 포매터: `.prettierrc*`, `rustfmt.toml`, `.editorconfig`, `biome.json`
   - 타입 체커: `tsconfig.json`, `mypy.ini`, `pyrightconfig.json`
   - CI: `.github/workflows/`, `.gitlab-ci.yml`, `Jenkinsfile` — CI 파이프라인에서 어떤 검증이 자동 실행되는지 확인
   - Pre-commit: `.pre-commit-config.yaml`, `.husky/`
4. 주요 진입점 확인 (`main.*`, `app.*`, `index.*`, `cmd/`, `src/main.*`)
5. 사용 언어/프레임워크 식별 (의존성 파일의 내용을 읽어서 주요 라이브러리 파악)

#### 2-2. 아키텍처 분석

**목표**: 코드가 어떻게 조직되어 있고, 컴포넌트 간 관계가 어떤지 파악한다.

1. **디렉토리 레이어링**: `src/`, `lib/`, `internal/`, `pkg/` 등의 구조에서 레이어 경계를 읽는다
2. **진입점 추적**: main/app 파일부터 시작하여 초기화 흐름을 따라간다 — DI 컨테이너, 라우터 등록, 미들웨어 체인 등
3. **도메인 경계**: 패키지/모듈 간 import 관계를 분석하여 도메인 경계를 식별한다
4. **설계 패턴**: 코드에서 반복되는 패턴을 식별한다 (Repository 패턴, CQRS, 이벤트 소싱, MVC, 클린 아키텍처 등)
5. **제약사항 추출**: 다음 소스에서 프로젝트의 제약사항을 수집한다:
   - 설정 파일의 하드코딩된 제한 (타임아웃, 최대 크기, rate limit 등)
   - 코드 내 `// TODO`, `// HACK`, `// FIXME`, `// WARN` 주석
   - README, CONTRIBUTING.md 등에 명시된 제약
   - 의존성 버전 고정(pinning)이나 호환성 제약
   - 환경 변수 목록 (`.env.example`, 설정 파일에서 추출)

#### 2-3. 영속성(Persistence) 분석

**목표**: 데이터가 어디에 어떻게 저장되고, 어떤 주의사항이 있는지 파악한다.

1. **데이터 저장소 식별**:
   - DB 연결 설정 파일 탐색 (`database.*`, `db.*`, `ormconfig.*`, `knexfile.*`, `alembic.ini`, `prisma/schema.prisma`, `drizzle.config.*`)
   - ORM/쿼리 빌더 사용 여부와 종류 (SQLAlchemy, Prisma, TypeORM, GORM, ActiveRecord 등)
   - 캐시 레이어 (Redis, Memcached 설정)
   - 파일 스토리지 (S3, GCS 등 설정)
2. **스키마/마이그레이션**:
   - 마이그레이션 디렉토리 탐색 (`migrations/`, `db/migrate/`, `alembic/versions/`, `prisma/migrations/`)
   - 현재 스키마 구조 파악 (모델/엔티티 정의 파일 읽기)
   - 마이그레이션 전략 확인 (자동 생성 vs 수동 작성)
3. **주의사항 추출**:
   - 트랜잭션 경계가 어떻게 관리되는지
   - N+1 쿼리 방지 패턴 사용 여부
   - 소프트 삭제 vs 하드 삭제 전략
   - 데이터 시딩/초기화 스크립트 존재 여부

#### 2-4. API/엔드포인트 분석 (웹 서비스인 경우)

**목표**: 어떤 엔드포인트가 있고, 서로 어떻게 연관되는지 파악한다.

이 단계는 웹 프레임워크가 감지된 경우에만 수행한다 (Express, FastAPI, Gin, Rails, Spring, Django, Next.js 등).

1. **라우터/컨트롤러 탐색**:
   - 프레임워크별 라우트 정의 패턴을 검색한다:
     - Express/Koa: `router.get`, `router.post`, `app.use`
     - FastAPI: `@app.get`, `@router.post`, `APIRouter`
     - Django: `urlpatterns`, `path(`, `re_path(`
     - Rails: `routes.rb`, `resources :`, `get '/'`
     - Spring: `@GetMapping`, `@PostMapping`, `@RequestMapping`
     - Go: `http.HandleFunc`, `r.GET`, `e.GET`
     - Next.js/Nuxt: `app/` 또는 `pages/` 디렉토리의 파일 기반 라우팅
   - OpenAPI/Swagger 스펙 파일이 있으면 우선 활용한다 (`openapi.yaml`, `swagger.json`)
2. **엔드포인트 목록 추출**: HTTP 메서드, 경로, 핸들러 함수명을 표로 정리한다
3. **엔드포인트 간 연관성 분석**:
   - 리소스 기반 그룹핑 (같은 리소스에 대한 CRUD 엔드포인트)
   - 미들웨어 체인 (인증, 로깅, rate limiting 등이 어떤 라우트에 적용되는지)
   - 엔드포인트 간 호출 관계 (내부 API 호출, 이벤트 발행 등)
4. **인증/인가 흐름**: 어떤 엔드포인트가 보호되고, 어떤 미들웨어/데코레이터가 사용되는지

#### 2-5. 문서 후보 결정

저장소 분석 결과를 바탕으로, 아래 후보 목록에서 **실제로 의미 있는 문서만** 선별한다. 작은 프로젝트에 10개 문서를 만들 필요는 없다.

### 3단계: 갱신 모드 (기존 지식 베이스가 있을 때)

1. 기존 문서를 읽고 현재 코드와의 차이를 파악한다
2. `git log --since` 등을 활용하여 마지막 문서 업데이트 이후의 변경사항을 확인한다
3. 변경된 부분만 문서에 반영한다 — 전체를 다시 쓰지 않는다
4. 새로운 도메인이나 모듈이 추가되었다면 해당 문서를 새로 생성한다
5. 삭제된 기능에 대한 문서는 제거하거나 아카이브한다

## 문서 구조

### 핵심 문서 (루트 레벨)

프로젝트 전반을 조망하는 문서들. 항상 최신 상태를 유지해야 한다.

| 파일 | 용도 | 생성 조건 |
|------|------|-----------|
| `ARCHITECTURE.md` | 도메인 맵, 패키지 레이어링, 시스템 전체 구조 | **항상 생성** |
| `DESIGN.md` | 핵심 설계 결정, 기술 선택 근거, 트레이드오프 | 설계 결정이 비자명한 경우 |
| `FRONTEND.md` | 프론트엔드 아키텍처, 컴포넌트 구조, 상태 관리 | 프론트엔드가 있는 경우 |
| `SECURITY.md` | 인증/인가 흐름, 보안 경계, 민감 데이터 처리 | 인증/보안 로직이 있는 경우 |
| `RELIABILITY.md` | 에러 처리 전략, 재시도, 서킷 브레이커, 모니터링 | 분산 시스템이거나 신뢰성이 중요한 경우 |
| `PRODUCT_SENSE.md` | 비즈니스 도메인 용어집, 사용자 흐름, 핵심 유스케이스 | 도메인 로직이 복잡한 경우 |
| `QUALITY_SCORE.md` | 도메인/레이어별 품질 등급, 기술 부채 현황 | 규모가 큰 프로젝트 |
| `PLANS.md` | 진행 중/완료된 계획 색인, 실행 계획 포인터 | 실행 계획을 관리하는 경우 |

### 누적형 문서 (docs/ 하위)

시간이 지나면서 쌓이는 문서들. 목차(index.md)를 통해 탐색한다.

**규칙: docs/ 하위의 모든 디렉토리에는 반드시 `index.md`를 포함한다.**

index.md는 토큰 절감의 핵심 장치다. 에이전트가 특정 정보를 찾을 때 디렉토리 내 모든 문서를 읽는 대신, index.md만 먼저 읽으면 어떤 문서에 원하는 내용이 있는지 판단할 수 있다. 따라서 index.md에는 각 문서의 제목뿐 아니라 **핵심 내용 요약**(1줄)을 반드시 포함하여, 에이전트가 해당 문서를 열어볼 필요가 있는지 사전에 판단할 수 있게 한다. 문서가 1개뿐이더라도 index.md를 생성한다 — 이후 문서가 추가될 때 일관된 탐색 경로를 보장하기 위함이다.

```
docs/
├── design-docs/          # 설계 문서 (ADR 등)
│   ├── index.md          # 목차: 제목, 상태, 날짜
│   └── <topic>.md
├── exec-plans/           # 실행 계획
│   ├── index.md          # 진행 중/완료 계획 색인
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── product-specs/        # 제품 스펙
│   ├── index.md          # 기능별 스펙 색인
│   └── <feature>.md
├── generated/            # 자동 생성 문서 (DB 스키마 등)
│   ├── index.md          # 생성된 문서 목록과 생성 방법
│   └── db-schema.md
└── references/           # 외부 참조 자료 (LLM용 문서 등)
    ├── index.md          # 참조 자료 목록과 용도
    └── <library>-llms.txt
```

| 디렉토리 | 용도 | 생성 조건 |
|-----------|------|-----------|
| `design-docs/` | 아키텍처 결정 기록(ADR), 설계 문서 | 주요 설계 결정이 여러 개인 경우 |
| `exec-plans/` | 실행 계획, 기술 부채 추적 | 복잡한 마이그레이션이나 대규모 변경이 있는 경우 |
| `product-specs/` | 기능별 제품 스펙 | 제품 요구사항이 복잡한 경우 |
| `generated/` | 스크립트로 자동 생성되는 문서 | DB 스키마 등 자동 추출 가능한 정보가 있는 경우 |
| `references/` | 외부 라이브러리/도구 참조 문서 | 에이전트가 자주 참조할 외부 문서가 있는 경우 |

## 문서 작성 규칙

### 목차(TOC) 형식

모든 핵심 문서의 상단에 목차를 포함한다. 에이전트가 파일의 처음 몇 줄만 읽고도 어떤 섹션이 있는지 파악할 수 있어야 한다.

```markdown
# 문서 제목

> 한 줄 요약: 이 문서가 다루는 범위

## 목차
- [섹션 1](#섹션-1): 간략 설명
- [섹션 2](#섹션-2): 간략 설명
- ...

---

## 섹션 1
...
```

### ARCHITECTURE.md 필수 구조

```markdown
# Architecture

> 한 줄 요약

## 목차
- [시스템 개요](#시스템-개요)
- [기술 스택](#기술-스택)
- [개발 환경](#개발-환경)
- [디렉토리 구조](#디렉토리-구조)
- [도메인 맵](#도메인-맵)
- [레이어 구조](#레이어-구조)
- [데이터 흐름](#데이터-흐름)
- [영속성](#영속성)
- [API 엔드포인트](#api-엔드포인트) *(웹 서비스인 경우)*
- [외부 연동](#외부-연동)
- [제약사항](#제약사항)

---

## 시스템 개요
프로젝트의 목적, 주요 기능, 사용자를 1-2 문단으로 설명

## 기술 스택
언어, 프레임워크, 주요 라이브러리를 표로 정리

## 개발 환경
패키지 매니저, 빌드 도구, 린터/포매터, CI 파이프라인, pre-commit 훅 등 개발 워크플로우에 필요한 도구 체인을 정리

## 디렉토리 구조
주요 디렉토리의 역할을 트리 형태로 표현 (깊이 2-3 수준)

## 도메인 맵
비즈니스 도메인의 경계와 관계를 설명

## 레이어 구조
아키텍처 레이어(프레젠테이션, 비즈니스, 데이터 등)와 의존성 방향

## 데이터 흐름
주요 데이터가 시스템을 통해 흐르는 경로

## 영속성
DB, ORM, 마이그레이션 전략, 캐시 레이어, 스토리지. 트랜잭션 관리 방식과 주의사항 포함

## API 엔드포인트
*(웹 서비스인 경우에만 포함)*
HTTP 메서드, 경로, 핸들러를 표로 정리. 리소스별 그룹핑, 미들웨어 적용 범위, 인증 요구사항 포함

## 외부 연동
외부 API, DB, 메시지 큐 등과의 연결 지점

## 제약사항
환경 변수 요구사항, 하드코딩된 제한값, 호환성 제약, 알려진 한계점
```

### 누적형 문서의 index.md 형식

```markdown
# <카테고리> 목차

| 제목 | 요약 | 상태 | 날짜 | 파일 |
|------|------|------|------|------|
| 인증 시스템 재설계 | JWT에서 세션 기반으로 전환, RBAC 도입 | 완료 | 2025-01-15 | [auth-redesign.md](./auth-redesign.md) |
| API 버전닝 전략 | URL 경로 기반 버전닝, v1/v2 병행 운영 방안 | 진행중 | 2025-02-01 | [api-versioning.md](./api-versioning.md) |
```

요약 컬럼은 에이전트가 문서를 열지 않고도 내용을 판단할 수 있는 핵심 장치다. 1줄 이내로 해당 문서의 핵심 내용을 기술한다.

### 작성 원칙

1. **사실 기반**: 코드에서 확인한 내용만 기술한다. 추측은 명시적으로 `[추정]`으로 표기한다
2. **파일 경로 포함**: 주요 설명에는 관련 소스 파일 경로를 함께 기록한다
3. **간결함**: 같은 내용을 반복하지 않는다. 다른 문서를 참조할 때는 링크를 사용한다
4. **한국어 작성**: 모든 문서를 한국어로 작성한다. 기술 용어는 원어를 병기할 수 있다 (예: "서킷 브레이커(Circuit Breaker)")
5. **에이전트 친화적**: 에이전트가 파일을 부분적으로 읽을 수 있도록 섹션을 명확히 구분한다

## 실행 시 참고사항

- 분석 대상이 모노레포인 경우, 각 패키지/앱을 별도 도메인으로 취급하여 ARCHITECTURE.md에서 전체 관계를 설명하고, 필요시 하위 README나 문서를 생성한다
- 기존 README.md나 CLAUDE.md가 있다면 내용을 존중하고 중복을 피한다
- 지식 베이스는 코드와 함께 버전 관리되어야 한다. 커밋 메시지에 문서 변경 범위를 명시한다
- 대규모 저장소에서는 Agent 도구를 활용하여 여러 디렉토리를 병렬로 분석한다
