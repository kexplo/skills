# Forge — 설계부터 구현까지

## 개요

Forge는 사용자의 구현 요구사항을 에이전트 파이프라인으로 처리하여, PRD 작성부터 최종 QA까지 전체 개발 사이클을 완수하는 스킬이다.

큰 규모의 프로젝트는 아키텍처 단계에서 **빌딩 블록**으로 분해하고, 각 블록을 깊이 우선(depth-first)으로 구현한 뒤 점진적으로 통합한다. 이를 통해 컨텍스트 윈도우 한계를 극복하고, 블록 단위의 검증으로 통합 리스크를 줄인다.

각 단계는 전문 서브에이전트가 담당하며, 단계별 산출물은 `docs/forge/` 디렉토리에 Markdown으로 저장된다.

## 파이프라인 흐름

```
[사용자 요구사항]
       │
       ▼
  ┌─────────────┐
  │ 1. PRD 에이전트 │ ── 사용자와 인터랙티브 Q&A → docs/forge/prd.md
  └──────┬──────┘
         │ 사용자 확인
         ▼
  ┌───────────────────────┐
  │ 2. 아키텍처 에이전트     │ → docs/forge/architecture.md
  │    (빌딩 블록 분해 포함)  │    블록별 의존성, 구현 순서, 핵심 블록 표시
  └──────┬────────────────┘
         ▼
  ┌──────────────────────────┐
  │ 3. Devil's Advocate 리뷰  │ → docs/forge/architecture-review.md
  └──────┬───────────────────┘
         │ 아키텍처 에이전트가 리뷰를 검토 후 선택적 수용
         ▼
  ┌──────────────────┐
  │ 2b. 아키텍처 수정  │ → docs/forge/architecture.md (갱신)
  └──────┬───────────┘
         ▼
  ┌──────────────────────────────────────────┐
  │ 4. 블록별 깊이 우선 구현                     │
  │                                            │
  │  블록 A (핵심) ─┐                           │
  │    구현+테스트    ├→ 핵심 블록 리뷰 → 수정     │
  │                  │                          │
  │  블록 B ─────────┤                          │
  │    구현+테스트    ├→ A+B 통합 테스트          │
  │                  │                          │
  │  블록 C ─────────┘                          │
  │    구현+테스트   → A+B+C 통합 테스트          │
  └──────┬───────────────────────────────────┘
         ▼
  ┌──────────────────────────┐
  │ 5. 전체 구현 리뷰          │ → docs/forge/implementation-review.md
  └──────┬───────────────────┘
         │ 구현 에이전트가 리뷰를 검토 후 선택적 수용
         ▼
  ┌──────────────────┐
  │ 4b. 구현 수정     │ → 소스코드 + 테스트 (갱신)
  └──────┬───────────┘
         ▼
  ┌──────────────────┐
  │ 6. QA 엔지니어    │ → docs/forge/qa-report.md
  └──────┬───────────┘
         ▼
  ┌──────────────────┐
  │ 7. 최종 리포트    │ → docs/forge/final-report.md
  └──────┬───────────┘
         │ 사용자 확인
         ▼
  [완료]
```

## 서브에이전트 스폰 가이드

각 서브에이전트를 Agent tool로 스폰할 때 반드시 다음을 지정한다:

- **`mode: dontAsk`**: 서브에이전트가 파일 생성, 명령 실행 등을 사용자 승인 없이 수행할 수 있게 한다. 백그라운드 에이전트는 사용자와 대화할 수 없으므로, 이 모드가 없으면 권한 거부로 파이프라인이 중단된다.
- **필요한 컨텍스트를 프롬프트에 모두 포함**: 서브에이전트는 이전 단계의 파일을 직접 읽을 수 있지만, 어떤 파일을 읽어야 하는지 명시적으로 안내한다.

```
Agent tool 호출 예시:
- mode: dontAsk
- prompt: (에이전트 지침 + 필요한 컨텍스트)
```

## 실행 절차

### 0단계: 준비

사용자의 요구사항으로부터 간결한 태스크명을 도출하고, 산출물 디렉토리를 생성한다.

**산출물 경로**: `docs/forge/<YYYYMMDD>-<태스크명>/`
- 날짜는 실행 시점 기준 (예: `20260315`)
- 태스크명은 영문 kebab-case, 2~4단어 (예: `todo-api`, `notification-system`, `csv-cli-tool`)
- 예시: `docs/forge/20260315-todo-api/`

이후 모든 단계에서 `docs/forge/` 경로는 이 디렉토리를 가리킨다. 서브에이전트에게 전달할 때도 이 절대 경로를 명시한다.

### 1단계: PRD 작성

이 단계는 사용자와의 인터랙티브 대화가 필요하므로 **메인 세션에서 직접** 수행한다.

1. 사용자의 요구사항을 분석하여 부족한 정보를 식별한다.
2. AskUserQuestion 도구를 사용하여 부족한 정보를 질문한다. 질문은 구체적이고 선택지가 있는 형태로 구성한다.
3. 답변을 받지 못하거나 사용자가 "모르겠다"고 한 부분은 `?`로 표시한다.
4. 수집한 정보를 바탕으로 `agents/prd-agent.md`의 지침을 따라 PRD를 작성한다.
5. `docs/forge/prd.md`에 저장한다.
6. 사용자에게 PRD를 요약 제시하고 확인을 받는다. 수정 요청이 있으면 반영한다.

### 2단계: 아키텍처 설계 + 빌딩 블록 분해

PRD가 확정되면 아키텍처 에이전트를 서브에이전트로 스폰한다.

```
Agent tool 호출:
- mode: dontAsk
- prompt: agents/architecture-agent.md의 지침 + PRD 내용 전달
- 산출물: docs/forge/architecture.md
```

서브에이전트에게 전달할 컨텍스트:
- `docs/forge/prd.md` 전체 내용
- 프로젝트의 기존 코드베이스 구조 (있는 경우)
- 사용 중인 기술 스택 정보

이 단계에서 아키텍처 에이전트는 **빌딩 블록 분해**를 포함한다:
- 프로젝트를 독립적으로 구현/테스트 가능한 블록으로 분해
- 블록 간 의존성 그래프와 구현 순서 정의
- 각 블록의 핵심 여부(★) 표시 — 핵심 블록은 개별 리뷰 대상
- 의존성이 없는 블록들의 병렬 구현 가능 여부 판단

### 3단계: 아키텍처 리뷰

아키텍처 문서가 완성되면 Devil's Advocate 리뷰어를 서브에이전트로 스폰한다.

```
Agent tool 호출:
- mode: dontAsk
- prompt: agents/devils-advocate-reviewer.md의 지침 + 리뷰 대상: architecture
- 입력: docs/forge/prd.md + docs/forge/architecture.md
- 산출물: docs/forge/architecture-review.md
```

리뷰어는 블록 분해의 적절성도 함께 검토한다:
- 블록 경계가 적절한지
- 의존성 그래프에 누락이 없는지
- 핵심 블록 선정이 타당한지

### 2b단계: 아키텍처 수정

리뷰 결과를 반영하여 아키텍처를 수정한다. 아키텍처 에이전트를 다시 서브에이전트로 스폰한다.

```
Agent tool 호출:
- mode: dontAsk
- prompt: agents/architecture-agent.md의 수정 지침 + 리뷰 내용 전달
- 핵심 원칙: 리뷰를 그대로 수용하지 않고, 각 지적의 타당성을 기술적으로 검토하여 선택적으로 수용
- 입력: docs/forge/architecture.md + docs/forge/architecture-review.md
- 산출물: docs/forge/architecture.md (갱신) + docs/forge/architecture-revision-notes.md (수용/거부 근거)
```

### 4단계: 블록별 깊이 우선 구현

확정된 아키텍처의 빌딩 블록 계획에 따라 **블록 단위로** 구현을 진행한다.

#### 구현 순서 결정

아키텍처의 빌딩 블록 섹션에서 정의한 순서를 따른다:
1. 의존성이 없는 기반 블록부터 시작
2. 의존성 그래프를 따라 깊이 우선으로 진행
3. 아키텍처가 병렬 구현 가능으로 표시한 블록들은 동시에 서브에이전트를 스폰할 수 있음

#### 각 블록의 구현 사이클

블록마다 다음 사이클을 실행한다:

**a) 블록 구현 (서브에이전트)**
```
Agent tool 호출:
- mode: dontAsk
- prompt: agents/implementation-agent.md의 지침
- 입력: docs/forge/prd.md + docs/forge/architecture.md + 현재 블록 이름
       + 이전에 구현된 블록의 소스코드 경로 (통합에 필요한 경우)
- 산출물: 블록의 소스코드 + 단위 테스트
```

구현 에이전트는:
- 해당 블록의 아키텍처 명세만 집중하여 구현
- 이전 블록과의 인터페이스를 올바르게 연결
- 아키텍처에서 정의한 해당 블록의 검증 방법을 모두 테스트로 구현
- 이전 블록이 있다면 통합 테스트도 작성

**b) 핵심 블록 리뷰 (조건부)**

아키텍처에서 핵심(★)으로 표시된 블록에 한해:
```
Agent tool 호출:
- mode: dontAsk
- prompt: agents/devils-advocate-reviewer.md의 지침 + 리뷰 대상: block-implementation
- 입력: 해당 블록 소스코드 + 테스트코드 + 아키텍처 중 해당 블록 명세
- 산출물: docs/forge/block-reviews/[블록명]-review.md
```

리뷰 후 구현 에이전트가 선택적으로 수용하여 수정한다.

**c) 통합 확인**

블록 구현 완료 후 기존 블록과의 통합 테스트를 실행한다. 실패하면 해당 블록을 수정한다.

#### 블록 구현 진행 상황 관리

메인 세션에서 각 블록의 진행 상황을 추적한다:
- 블록별 상태: 대기 → 구현 중 → 구현 완료 → 리뷰 중 → 통합 완료
- 블록 완료 시마다 사용자에게 진행 상황을 간략히 보고

### 5단계: 전체 구현 리뷰

모든 블록의 구현과 통합이 완료되면 전체 코드베이스에 대해 Devil's Advocate 리뷰를 수행한다.

```
Agent tool 호출:
- mode: dontAsk
- prompt: agents/devils-advocate-reviewer.md의 지침 + 리뷰 대상: implementation
- 입력: docs/forge/prd.md + docs/forge/architecture.md + 전체 소스코드/테스트코드
- 산출물: docs/forge/implementation-review.md
```

이 리뷰는 블록 간 통합 품질, 일관성, 전체적인 코드 품질에 초점을 둔다.

### 4b단계: 구현 수정

리뷰 결과를 반영하여 구현을 수정한다. 구현 에이전트를 다시 서브에이전트로 스폰한다.

```
Agent tool 호출:
- mode: dontAsk
- prompt: agents/implementation-agent.md의 수정 지침 + 리뷰 내용 전달
- 핵심 원칙: 리뷰를 그대로 수용하지 않고, 각 지적의 타당성을 기술적으로 검토하여 선택적으로 수용
- 입력: 기존 소스코드 + docs/forge/implementation-review.md
- 산출물: 소스코드 (갱신) + docs/forge/implementation-revision-notes.md (수용/거부 근거)
```

### 6단계: QA 검수

최종 QA 엔지니어를 서브에이전트로 스폰한다.

```
Agent tool 호출:
- mode: dontAsk
- prompt: agents/qa-engineer.md의 지침
- 입력: docs/forge/prd.md + docs/forge/architecture.md + 소스코드 + 테스트코드
- 산출물: docs/forge/qa-report.md
```

QA에서 심각한 문제가 발견되면:
1. 문제 내용을 사용자에게 보고한다.
2. 사용자 확인 후 해당 블록의 구현 단계로 돌아가 수정한다.

### 7단계: 최종 리포트 작성

모든 단계가 완료되면 `docs/forge/final-report.md`에 최종 리포트를 작성하고, 사용자에게 제시하여 확인을 받는다.

최종 리포트는 사용자가 전체 파이프라인의 결과를 한눈에 파악하고 판단할 수 있도록 구성한다. 리포트 작성 후 사용자에게 요약을 제시하고, 상세 내용은 파일을 참조하도록 안내한다.

#### 최종 리포트 구조

```markdown
# Forge 최종 리포트: [프로젝트/기능명]

## 1. 요약
- **최종 상태**: 완료 / 조건부 완료
- **QA 판정**: PASS / CONDITIONAL PASS / FAIL
- **빌딩 블록**: N개 (핵심 N개)
- **테스트**: 전체 N건, 통과 N건

## 2. 요구사항 달성 현황
PRD의 필수 기능별 구현 상태를 체크리스트로 정리:
- [x] 기능 1: 구현 완료, 테스트 통과
- [x] 기능 2: 구현 완료, 테스트 통과
- [ ] 기능 3: 미구현 (사유: ?)

## 3. 아키텍처 요약
- 선택한 아키텍처 스타일과 핵심 설계 결정
- 블록 분해 구조 (간략한 다이어그램)

## 4. 블록별 구현 결과
| 블록명 | 핵심 | 상태 | 단위 테스트 | 통합 테스트 | 리뷰 결과 |
|--------|------|------|-----------|-----------|----------|
| 블록 A | ★    | 완료 | 15/15     | 3/3       | 수정 반영 |
| 블록 B |      | 완료 | 8/8       | 2/2       | -        |

## 5. 리뷰 반영 요약
### 아키텍처 리뷰
- 수용: N건, 거부: N건
- 주요 수정 사항 요약

### 핵심 블록 리뷰 (해당시)
- 블록별 주요 지적과 반영 결과

### 전체 구현 리뷰
- 수용: N건, 거부: N건
- 주요 수정 사항 요약

## 6. QA 결과
- 테스트 커버리지 요약
- 발견된 주요 이슈와 해결 상태
- 자동화 테스트의 한계 및 추가 검증 필요 사항

## 7. 산출물 목록
### 문서
| 파일 | 설명 |
|------|------|
| docs/forge/prd.md | PRD |
| docs/forge/architecture.md | 아키텍처 설계 |
| ... | ... |

### 소스코드
| 경로 | 설명 |
```

## 규모별 동작 방식

아키텍처 에이전트가 프로젝트 규모를 판단하여 파이프라인을 조절한다:

- **소규모** (블록 1~2개): 블록 분해 없이 기존 방식대로 단일 구현. 사실상 블록이 하나인 것과 동일.
- **중규모** (블록 3~5개): 블록별 순차 구현, 핵심 블록만 개별 리뷰.
- **대규모** (블록 6개 이상): 블록별 구현, 의존성 없는 블록은 병렬 구현 고려, 핵심 블록 개별 리뷰.

규모 판단은 아키텍처 에이전트의 재량이며, 블록 수는 대략적인 가이드라인이다.

## 에이전트 레퍼런스

각 서브에이전트의 상세 지침은 `agents/` 디렉토리에 있다. 서브에이전트를 스폰할 때 해당 파일을 읽어서 프롬프트에 포함시킨다.

- `agents/prd-agent.md` — PRD 작성 지침
- `agents/architecture-agent.md` — 아키텍처 설계 + 빌딩 블록 분해 지침
- `agents/devils-advocate-reviewer.md` — Devil's Advocate 리뷰 지침
- `agents/implementation-agent.md` — TDD 기반 구현 지침 (블록별 구현 포함)
- `agents/qa-engineer.md` — QA 검수 지침

## 주의사항

- 모든 산출물과 커뮤니케이션은 한국어로 작성한다.
- 각 서브에이전트는 독립적으로 동작하므로, 필요한 컨텍스트(PRD, 아키텍처 등)를 반드시 프롬프트에 포함시켜야 한다.
- Devil's Advocate 리뷰 후 수정 단계에서는 리뷰를 무비판적으로 수용하지 않는다. 각 지적에 대해 기술적 타당성을 검토하고, 수용/거부 근거를 명시한다.
- QA 단계에서 심각한 결함이 발견되면 구현 단계로 회귀할 수 있다. 이때 사용자에게 상황을 설명하고 확인을 받는다.
- 기존 코드베이스가 있는 프로젝트에서는 기존 코드 스타일과 패턴을 존중한다.
- 블록별 구현 시, 각 블록의 서브에이전트에게 이전에 완료된 블록의 코드를 읽을 수 있는 경로를 반드시 안내한다.
